第一章 风格style

概述

  • 代码风格因人而异
  • 有争议的风格
    • 遵从你所在组织的规范
    • 与同一个文件中其他代码风格保持一致
    • 与同一项目里其他文件的风格尽量保持一致
  • 无争议的风格：命名风格，表达式和语句的风格，一致性和习惯用法，函数宏定义，避免神秘的数，注释风格等

程序的风格

命名的风格

  • 名字要准确的表达用途
  • 名字不是越长越好
    • 全局变量可以用具有描述意义的长名字
    • 局部变量用短名字会显得程序更简洁
  • 类的成员的名字，不要再啰嗦重复类名
  • 用动词或者动词+名次定义函数名
  • 返回布尔值的函数名应该是一个命题，通常使用is作为函数名中的动词
  • 名字尽可能简单明了，不要用“否定之否定”

表达式和语句

  • 使用表达式的自然形式，即常见的易于理解的形式
  • 利用括号排除歧义，即使语法上不要求括号，也建议加上括号以便提高代码可读性
  • 分解复杂的表达式

一致性和习惯用法

  • 使用一致的缩进和花括号风格(例如两行以上一定加花括号)
  • 使用大多数人最习惯的用法(for循环)
  • 用else if处理多路选择，一个判断尽可能接近它所对应的动作

函数宏

  • 避免宏定义参数中写很多东西。函数宏的缺点，直接替换，导致用到两次的参数被替换两次，执行两次，可能直接导致错误
  • 函数宏的缺点，直接替换，导致用到两次的参数会被替换两次，执行两次，也可能导致性能错误
  • 如果一定要用函数宏，不要吝啬括号

总结

  • C++ 尽可能使用inline函数
  • Java根本不支持函数宏

神秘的数

  • 给所有数字起一个名字
  • 直接使用字符形式的常量，不要使用它的整数形式
  • 利用语言去计算对象的大小，从而减少代码修改负担，提高代码简洁性，可以定义一个通用的宏
  • 尽可能让编译器来计算数值

注释的风格

  • 注释与代码要始终保持一致
  • 不要写没用的注释
  • 用好的命名代替注释
  • 用注释自动生成文档

设计与实现

算法和数据结构

查找

  • 数组下标访问
    • 最简单，最快，最常用的“算法”
    • 应用条件
      • 要查询的数据的索引都是不太大的整数
      • 这些整数的稀疏度不高
    • C标准库中，isdigit, isalpha等函数的实现，使用了数组下标直接访问
  • 顺序查找
    • 应用场景
      • 要查询的数据特别少，例如少于10条
      • 被查找数据的排列顺序不能自由设定，例如字符串中查找
      • 查询条件太复杂，以至于匹配条件的开销大于查找
      • 更新频繁，却很少查找
    • 有关字符串顺序查找的标准库
      • C: strchr, strstr
      • C++: std::find
      • Jaba: String.indexOf
  • 二分查找
    • 应用条件
      • 数据有序
      • 数据基本有序，可以很快调整成有序
      • 数据无需，但是排序后更新次数远小于查找次数
    • 高效，但应用条件苛刻

排序

  • 排序算法
    • 选择、冒泡、插入排序
    • 快速排序
    • 其他排序算法
  • 算法复杂度O记法

可增长数组

  • 为其中的指针申请内存

链表

  • std::list

树

散列表

  • 索引：整数
  • 散列数组大小：13
  • 散列算法
    • 字符串经过一个散列算法求得一个整数
    • 整数模除13得到散列数组下标
  • C++标准库：std::hash_map

如何选择

  • 要处理的数据有多少
  • 数据量在程序运行过程中会有多大的变化
  • 数据索引以及内容的分布有什么特点
  • 数据更新的频繁程度
  • 数据查询的频繁程序
  • 有现成的库可用吗
  • 程序的运行环境有什么限制

大作业

  • 设计实现一个脚本解释器

脚本

  • 为了特定目的，使用特定语法编写的一个文本文件，描述了一个流程，可以被一个解释器解释执行，以完成特定的功能
  • 文件->(Parser语法分析)->抽象语法树(AST)->(Interpreter)->展现(呼叫控制、媒体控制、自然语言分析->显示speak内容，输入客户意愿)

DSL和GPL

  • DSL 领域特定语言
  • GPL 通用编程语言

需求

  • 实现一个语音客服机器人
  • 实现一个自动应答逻辑
    • 处理用户的电话呼入
    • 借助媒体服务器播放提示音
    • 借助媒体服务器录音并将语音识别为文字
    • 借助自然语言分析引擎分析用户示意图
    • 根据用户意图决定下一步播放的提示音

方案一

  • 直接使用高级语言编程实现客服自动应答逻辑
  • 不同的逻辑编写不同的程序，需程序员不断参与

方案二

  • 一个解释器

Parser的实现

  • 脚本文件
  • ParseFile
  • 行
  • ParseLine
  • Tokens(词法元素) 标识符，字符串，整数等
  • 根据每行的第一个Token调用不同的处理函数

解释程序

  • 获取脚本语法树，创建执行环境
  • 当前Step置为entryStep
  • 循环针对当前step做
    • 执行Speak(语音合成，语音播放)
    • 如果本步骤是中介步骤，则结束循环，断开通话
    • 执行Listen(录音，语音识别，自然语言理解)
    • 获得下一个StepId
    • 将当前Step置为刚才获得的StepId对应的Step

Interpreter的执行环境

  • 当两个用户同时和机器人对话，解释器需要两个线程同时运行，每个线程服务一个用户
  • 同一份
    • 脚本文件
    • 脚本语法树(可以调用一次Parser形成，多个线程公用)
  • 两份
    • 表示姓名和账单的变量表
      • 用什么数据结构，如何构建
    • 脚本的当前执行状态

接口

程序与程序之间的接口

  • 库函数接口
    • 接口说明
      • 头文件
      • 联机main
      • 接口说明文档
      • 编程指南
    • 信息隐藏
      • 隐藏复杂的内部实现细节，暴露给调用者的概念尽可能的少
    • 资源管理
      • 库函数不能偷偷申请资源，如果申请应该负责释放
      • 谁申请谁释放
    • 错误处理
      • 库函数的接口应该安静地返回不同的返回值，让调用者根据返回值做不同的处理
    • 兼容性
      • 修改时保证原先的功能还可以实现
    • 回调函数 (???)
    • 库函数接口的缺点
      • 应用程序和库的代码在同一个进程内运行，共享相同的地址空间。如果程序运行出现一些神秘的BUG，往往难以定位责任
        • 因此，库函数接口适合的应用场合
          • 可靠的库
          • 个人项目
          • 小团队内项目
          • 合作良好的团队间项目
        • 库函数接口不适合的应用场景
          • 使用不熟悉的团队的不可靠的库
      • 应用程序和库的代码在同一个进程内运行，如果应用程序对实时性要求较高，而库函数内可能有较长时间的阻塞操作，会引起严重的性能问题
        • 解决方案 (???)
          • 启动单独线程运行库函数
          • 库函数提供异步接口
  • 共享数据接口
    • 接口说明(数据库表说明)
      • 每个表的名称、含义
      • 每个字段的名称，类型，取值范围，含义
      • 表和字段的关联逻辑
    • 使用共享数据接口有哪些缺点
      • 一方修改时必须通知其他使用者
    • 如何规避这些缺点导致的问题
      • ->通信协议接口
  • 通信协议接口
    • 接口说明(通信协议说明)
      • 承载协议
      • 应用协议
      • 每个消息的说明
      • 消息的每个字段的说明
    • 团队内部的两个模块可以用共享数据
    • 不同团队之间用通信协议来做
    • 定义通信协议接口的三个要点
      • 选择数据传输承载协议
      • 定义接口的数据格式
      • 定义接口双方的处理行为
    • 承载协议
      • 基于UDP(简单，但保证不了可靠性，当接口对可靠性要求不高的时候用UDP)
      • 基于TCP(可靠)
      • 基于更高层面的承载协议
        • HTTP，目前应用最广泛的一种无状态应答式接口承载协议
        • WebSocket，由HTTP衍生，面向流数据
        • RESTFUL，基于HTTP，一种特定的简约风格的接口设计
        • MQTT，广泛应用于物联网领域的一种轻量级发布/订阅模式协议
        • ZeroMQ，一个基于消息队列的多线程网络库，囊括了4种最通用的通信模型
        • KAFKA，基于消费者/生产者模式的实时消息中间件
    • 数据格式定义
      • 直接描述二进制格式，对消息的每一个参数的格式和长度进行描述
      • 特殊文本格式，使用BNF范式进行描述
      • 基于某种通用格式，使用这种通用格式的特定描述方法进行描述
        • ASN.1: Abstract Syntax Notation 1，抽象文法记法1
        • ProtoBuf
        • XML
        • JSON
      • 范例
        • 二进制格式的Diameter协议
        • 纯文本格式的SIP协议 RFC3261
        • 基于json的百度语音识别接口
        • 基于ASN.1描述的SNMP协议 RFC1157
        • ProtoBuf
    • 接口的兼容性
      • 新增的内容不能改变原来的内容

人与程序之间的接口

  • 命令行接口
    • 优势
      • 效率：批处理，复用，低带宽占用，低主机资源占用
      • 稳定：50年前的unix命令现在仍然可用
      • 低成本：对于不经常用的，面向小众(开发运维)的用户接口，低成本提供大量强大功能
    • 运维人员
      • stdin/stdout
      • about 查看版本
      • ls 查看已经加载的客服逻辑脚本，查看脚本语法树在内存中的加载情况
      • vm 查看解释器执行环境的情况，当前变量表，当前Step
      • st 查看客服脚本的执行统计信息
      • sip 查看与电话网之间的SIP链路情况，SIP消息的统计
      • ms 查看雨媒体服务器的链接情况，统计情况
      • se 与自然语言分析系统的连接情况，统计情况
      • db 数据库的连接情况
      • log 日志开关的打开/关闭
      • cfg 当前配置情况，重新加载配置
  • 配置文件
    • 规划配置项
      • 总的原则：提取不适合写死在程序中的数据作为配置文件中的配置项
      • 因为网络环境的变化而变化的数据
        • 本地的IP地址，端口
        • 目的地的IP地址，端口
      • 因为主机资源情况变化的数据
        • 线程池的大小
        • 消息队列的最大长度限制
      • 因为用户的偏好而变化的数据
        • 字符编码，UTF-8还是GBK？
        • 语言，中文还是英文？
    • 配置项生效
      • 有些重要的配置，修改后需要重启生效
      • 有的配置通过命令在线生效
        • 使用命令行单项修改配置，同步修改配置文件
        • 修改配置文件，使用命令行整体重新加载
      • 这两种配置需要谨慎区分，前者应该尽可能少
    • 配置文件的格式
      • 一个程序的多个配置文件的格式应该统一
        • 一开始就桂华好，选择一个合适的文件夹
        • 其他人在后期的程序修改中，不要私自添加其他格式的文件
      • 常用格式
        • 经典的ini格式,[xxx]分节，没节配置名字和取值
        • XML格式
        • JSON格式
      • 配置文件一定要允许加注释
        • 配置文件中的注释和程序中的注释一样重要
        • #开头的注释行
        • XML等格式本身支持的注释方式
    • 兼容性
      • 新增的配置项，必须是可选配置项
      • 不选时的默认行为，应该是上一个版本不包含此配置项的行为
  • 错误信息
    • 好的例子
      • cc编译器输出的错误信息
      • ssh连接失败输出的错误信息
      • git 命令使用错误时的输出信息
    • 输出完整准确的错误信息
  • 日志文件
    • 是了解程序的过往运行状况的主要参考，是排错的主要依据
    • 哪些内容需要写入日志文件
      • 错误信息
        • 在输出到标准输出，或者输出到前端页面的同时，需要同时写入日志文件
      • 重要的程序运行信息
        • 程序启动，关闭
        • 重要的通信链路连接断开
        • 重要的模块加载卸载

        • 详细的业务运行信息，模块之间的交互的借口信息

        • 开关可以使用命令行接口，实现不同的命令实现
          • trace db on 开始数据库sql语句的日志
          • trace db off 关闭数据库sql语句的日志
          • trace http on 开启http接口的原始消息写入日志
          • trace http off 关闭http接口的原始消息写入日志
    • 日志的要求
      • 一般要求
        • 详略得当，文字简练，完整，准确
        • 格式朴素，防止日志“内卷”
      • 每行日志的要素
        • 时间戳(可以精确到毫秒甚至微妙)
        • 发生的模块标识
        • 时间简要说明
        • 时间参数的详细列表(例如记录模块间交互的借口消息的日志)
      • 日志文件的大小控制
        • 日志文件不能积累太大，否则阅读，搜索，下载都是比较麻烦的事情
        • 跟就程序特点，可以设置每天，或者每周，每月一个日志文件
        • 要有自动备份，删除的机制，不要让日志占满了所有存储空间

排错

如何对待BUG

  • 首先，最好不要出BUG
    • 良好的设计
    • 良好的编程风格
    • 注意所采用的语言的编译器的告警
  • 其次，让BUG容易发现
    • 设计良好的用户界面(日志，错误信息，查看命令)
    • 边界条件的检查以及断言的使用
  • 再次，即使出现BUG，也要自己发现并解决
  • 再再次，自己没有发先，也尽可能让组织内部的测试人员发现
  • 一般应用程序运行出现BUG，首先，要99.99%确信
    • 不会是硬件的问题导致
    • 不会是网络传输的错误导致
    • 不会是操作系统的BUG导致
    • 不会是编译器的BUG导致
    • 不会是标准库的BUG导致
  • 程序运行出现BUG，不知为何，BUG又消失了，但是实际上
    • BUG肯定仍然存在
    • 而且这可能是一个比较难找的BUG
    • 尽可能让这个BUG复现，以便解决它
  • BUG越晚解决，解决它所需成本越高

Debug工具(debugger)

  • IDE集成的debuger，可以对程序执行单步跟踪，可以实时查看所有变量，但是，它们在处理一些较难的BUG的时候用途有限
  • linux和unix下都有基于命令行的调试工具gdb或者dbx，它功能上覆盖了IDE支持的所有功能
  • gdb或者dbx经常用于查看程序崩溃时留下的core文件
  • java程序执行遇到异常的时候，一般可以输出当前的堆栈信息有助于BUG的分析

DEBUG的一般步骤

  • 了解清楚BUG的表现
    • BUG的表现
      • 出现BUG的前置条件(主机、操作系统、程序版本、配置等等)
      • 出现BUG时的输入或操作
      • 出现BUG之后的现象(输出不正确？死循环？异常退出？)
    • BUG发现的时机，影响对BUG表现的了解
    • 了解这个BUG表现的普遍性和特殊性
  • 获取这个BUG的线索
    • BUG发生前后这段时间的日志
    • UNIX或Linux系统下，如果发生异常退出，获取core文件
    • 使用维护命令获取尽可能多的信息
  • 查阅相关版本的代码修改
    • 基于良好的习惯，所有所有代码的都应被版本管理工具妥善的管理
    • 如果版本n出现的BUG，可以diff版本n于版本n-1的差别，仔细阅读这些差别
  • 在自己的开发环境复现BUG
    • 开发环境往往无法与实际运行环境一致
    • 需要设计各种测试桩尽可能模拟实际运行环境
  • 在复现BUG的开发环境下分析
    • 可以在有所怀疑的地方，任意增加日志，增加断言，增加命令
    • 尽可能认真地增加这些东西，让它们以后还有用
    • 如果BUG消失了，那么刚才这个环境就是一个不能复现BUG的环境，毫无用处。恢复到以前，重新修改
  • 二分法缩小BUG的范围
    • (1-100行脚本+解释器)执行10000次=>发现内存泄漏
    • (1-50行脚本+解释器)执行10000次=>…
    • …….
  • 不能复现的BUG
    • 使用工具扫描代码(splint)，对工具给出的可以结果认真检查
    • 回顾自己的经验教训
      • 没初始化的变量
      • 函数返回值
      • 错误的指针应用
      • 多线程
    • 把你的BUG，你的代码讲给你周围的人听

常见内存错误

  • 使用删除后的指针 ？
  • 函数的局部变量，在函数退出以后就失效了
  • 检查一下参数的次序是否用错了
  • 检查一下，printf给出的格式和具体的参数类型是否匹配
  • 检查一下，数组访问的边界是否弄错
  • 检查一下，if的条件之后有没有不小心加分号
  • 检查一下定义了返回值的函数，是否真的每个分支都确实有返回值
  • 线程不安全函数，往往返回了相同的static存储空间的地址，后面的调用会覆盖前面的调用
    • 一般来说，c标准中所有线程不安全函数，都会对应有一个线程安全的版本，可以在需要的时候使用线程安全版本
  • 回调函数是否是另外一个线程调用的 ？
  • 含有析构函数的对象数组，删除的时候不要忘记使用delete[]，否则会导致内存泄露或崩溃
  • 不要忘记将基类析构函数定义为虚函数，否则会导致内存泄露
    • 如果不是虚函数，则析构派生类成员时，只会调用积累的需构函数
  • 含有虚函数的对象的序列化，需要注意“虚表指针”的问题 ？

测试

什么是测试

  • 测试只能证明程序存在BUG，并不能证明程序不存在BUG

谁来测试

  • 开发人员，应该首先做完善的测试，通过各种手段，发现程序的绝大部分BUG，并解决它们
  • 相对完整的开发机构，会有单独的测试部门，来负责系统测试
  • 有的软件产品，会允许一定范围内的试用用户进行测试
